12 Bidirectional communication for Kaleidoscope. With this plugin, one can expose a
13 set of commands via the Serial port, and allow the host to talk with the
14 keyboard - and vice versa. This plugin implements only the basic building
15 blocks, a framework other plugins can opt-in to.
16 
17 ## Using the plugin
18 
19 This plugin is **not** meant to be used by the end-user (apart from setting it
20 up to use plugin-provided hooks), but by plugin authors instead. As an end user,
21 please see the documentation of the plugins you use, for instructions on how to
22 hook them up with `Focus`!
23 
24 Nevertheless, the basic commands we implement with this plugin alone, are usable
25 like this:
26 
27 ```c++
28 #include <Kaleidoscope.h>
29 #include <Kaleidoscope-Focus.h>
30 
31 void setup () {
32  Serial.begin (9600);
33 
34  Kaleidoscope.use (&Focus);
35 
36  Kaleidoscope.setup ();
37 
38  Focus.addHook (FOCUS_HOOK_HELP);
39  Focus.addHook (FOCUS_HOOK_VERSION);
40 }
41 ```

43 ## Plugin methods
44 
45 The plugin provides the `Focus` object, which has the following method:
46 
47 ### `.addHook(FOCUS_HOOK (function, documentation))`
48 
49 > Adds a new hook to `Focus`. Hooks are called in order of registration, and
50 > they get the parsed command name as argument. If they handle the command, they
51 > shall return `true`, otherwise `false`. Once a command has been handled, it
52 > will not be given to other hooks.
53 >
54 > The hook function is responsible for reading the rest of the command, in
55 > whatever way it sees fit.
56 >
57 > The `documentation` argument is a string, used by the `help` command, and can
58 > be left empty, if no documentation is desired for the `function`.
59 
60 ## Focus commands
61 
62 The plugin ships with two (optional) hooks: `FOCUS_HOOK_VERSION`, and
63 `FOCUS_HOOK_HELP`, implementing the following two commands, respectively:
64 
65 ### `version`
66 
67 > Return the version of the firmware, the keyboard vendor & product, and the
68 > compile date.
69 
70 ### `help`
71 
72 > Return the list of commands the keyboard supports.

74 ## Wire protocol
75 
76 `Focus` uses a simple, textual, request-response-based wire protocol.
77 
78 Each request has to be on one line, anything before the first space is the
79 command part (if there is no space, just a newline, then the whole line will be
80 considered a command), everything after are arguments. The plugin itself only
81 parses until the end of the command part, argument parsing is left to the
82 various hooks. If there is anything left on the line after hooks are done
83 processing, it will be ignored.
84 
85 Responses can be multi-line, but most aren't. Their content is also up to the
86 hooks, `Focus` does not enforce anything, except a trailing dot and a newline.
87 Responses should end with a dot on its own line.
88 
89 Apart from these, there are no restrictions on what can go over the wire, but to
90 make the experience consistent, find a few guidelines below:
91 
92 * Commands should be namespaced, so that the plugin name, or functionality comes
93  first, then the sub-command or property. Such as `led.theme`, or `led.setAll`.
94 * One should not use setters and getters, but a single property command instead.
95  One, which when called without arguments, will act as a getter, and as a
96  setter otherwise.
97 * Namespaces should be lowercase, while the commands within them camel-case.
98 * Do as little work in the hooks as possible. While the protocol is human
99  readable, the expectation is that tools will be used to interact with the
100  keyboard.
101 * As such, keep formatting to the bare minimum. No fancy table-like responses.
102 * In general, the output of a getter should be copy-pasteable to a setter.
103 
104 These are merely guidelines, and there can be - and are - exceptions. Use your
105 discretion when writing Focus hooks.
