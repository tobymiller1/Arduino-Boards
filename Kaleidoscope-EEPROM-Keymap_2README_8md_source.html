12 While keyboards usually ship with a keymap programmed in, to be able to change
13 that keymap, without flashing new firmware, we need a way to place the keymap
14 into a place we can update at run-time, and which persists across reboots.
15 Fortunately, we have a bit of `EEPROM` on the keyboard, and can use it to store
16 either the full keymap (and saving space in the firmware then), or store an
17 overlay there. In the latter case, whenever there is a non-transparent key on
18 the overlay, we will use that instead of the keyboard default.
19 
20 In short, this plugin allows us to change our keymaps, without having to compile
21 and flash new firmware. It does so through the use of the [Focus][plugin:focus]
22 plugin.
23

26 ## Using the plugin
27 
28 Using the plugin is reasonably simple: after including the header, enable the
29 plugin, and let the `Layer` object know that we'll be using `EEPROMKeymap` to
30 figure out which keys to use. We can either use the `getKeyOverride` or the
31 `getKey` method, depending on whether we want to override, or fully replace the
32 built-in keymap. Then we need to set at most how many layers we want to store in
33 `EEPROM`, and that is about it.
34 
35 We can then update the keymap via [Focus][plugin:focus].
36 
37 ```c++
38 #include <Kaleidoscope.h>
39 #include <Kaleidoscope-EEPROM-Keymap.h>
40 #include <Kaleidoscope-EEPROM-Settings.h>
41 #include <Kaleidoscope-Focus.h>
42 
43 void setup() {
44  Serial.begin(9600);
45  Kaleidoscope.use(&EEPROMKeymap, &Focus);
46 
47  Kaleidoscope.setup();
48 
49  Focus.addHook(FOCUS_HOOK_KEYMAP);
50  Focus.addHook(FOCUS_HOOK_KEYMAP_TRANSFER);
51 
52  Layer.getKey = EEPROMKeymap.getKeyOverride;
53 
54  EEPROMKeymap.max_layers(1);
55  EEPROMSettings.seal();
56 }
57 ```

59 ## Plugin methods
60 
61 The plugin provides the `EEPROMKeymap` object, which has the following methods:
62 
63 ### `.max_layers(max)`
64 
65 > Tells the extension to reserve space in EEPROM for up to `max` layers. Can
66 > only be called once, any subsequent call will be a no-op.

68 ## Focus commands
69 
70 The plugin provides three `Focus` hooks: `FOCUS_HOOK_KEYMAP`, `FOCUS_HOOK_KEYMAP_LAYER`,
71 and `FOCUS_HOOK_KEYMAP_TRANSFER`. Together, they make the following commands
72 available, respectively:
73 
74 ### `keymap.map [codes...]`
75 
76 > Without arguments, displays the keymap currently in effect. Each key is
77 > printed as its raw, 16-bit keycode.
78 >
79 > With arguments, it stores as many keys as given. One does not need to set all
80 > keys, on all layers: the command will start from the first key on the first
81 > layer, and go on as long as it has input. It will not go past the layer set
82 > via the `.max_layers()` method.
83 
84 ### `keymap.layer LAYER [codes...]`
85 
86 > Without codes, prints the keymap for the given layer (zero-indexed).
87 > Prints each key as its raw 16-bit keycode.
88 
89 > With codes, stores them as the keymap for the given layer.
90 
91 ### `keymap.transfer LAYER`
92 
93 > Transfers the `LAYER` from the built-in memory of the keyboard into `EEPROM`
94 > storage.
95 >
96 > Useful mostly when one wants to remove the built-in keymap, and wants to
97 > easily transfer it into `EEPROM` first.
98 >
99 > This is generally not needed, and it is recommended to not enable this
100 > command, unless the feature this command implements is truly needed.
